# GEO Rewrite Skill

[🇨🇳 中文](./README.md) · [🇺🇸 English](./README.en.md) · [🇯🇵 日本語](#)

> 事実を置き換えるのではなく、あなたが既に持っている真実のコンテンツを、AI 時代に見えるようにする。

従来の記事を、生成 AI がより容易に**理解・信頼・抽出・引用・推奨**できるコンテンツへと再構築します。

> 完全なスキルファイル：[geo-rewrite-skill/SKILL.md](./geo-rewrite-skill/SKILL.md)（v1.2、25 章）。

## 核心理念

単なるリライトではありません。**AI が回答を生成する際に、あなたのコンテンツを優先的に引用させる**ことが目的です。

**DSS+E ゴールデンモデル** に基づいてリライト：

| 次元 | 意味 | 核心要件 |
|------|------|---------|
| **D**epth | 意味的深度 | 論理チェーン、因果推論、メカニズム分析の強化 |
| **S**upport | データサポート | ファクト密度の向上、2025〜2026年の最新データを優先 |
| **S**ources | 権威的情報源 | 複数ソースのクロス検証、機械検証可能なシグナル |
| **E**ntity | エンティティ権威 | ブランド/製品/用語を統一し、エンティティグラフを構築 |

## なぜ Write ではなく Rewrite なのか — GEO エコシステムガバナンスの倫理的基盤

GEO Rewrite は AI に代筆させるのではなく、あなたが既に持っている真実のコンテンツを、AI 時代に見えるようにします。

### AI コンテンツエコシステムは「信頼危機」に直面している

2026 年現在、インターネット上の長文コンテンツの約 27% が AI によって生成または支援されています。生成 AI モデルがこれらの AI 生成コンテンツを訓練用または引用用のデータとして使用すると、**「ハルシネーション再帰（Hallucination Recursion）」** が発生します——モデルが幻覚コンテンツを引用し、さらに多くの幻覚を生み出し、AI 引用エコシステム全体の信頼性が下方スパイラルに陥ります。

**GEO Rewrite の立場：新しい事実を作らず、既存の事実の発見可能性と引用可能性のみを強化する。**

### Rewrite vs Write：事実のアンカリングと幻覚生成の本質的な違い

| | AI Write（直接生成） | GEO Rewrite（リライト） |
|---|---|---|
| 事実の出所 | モデルの訓練データ、追跡不能、捏造リスクあり | 原著作者の真実の情報、追跡可能 |
| データの真正性 | 具体的な数字を捏造する可能性（幻覚） | 原文の既存データのみを構造化、新しい数値は追加しない |
| ブランドエンティティ | 混同、捏造、誤帰属の可能性 | 原文エンティティを統一強化、逸脱ゼロ |
| 倫理的リスク | 幻覚が引用エコシステムを汚染 | 新たな幻覚ゼロ |
| AI 引用価値 | 次ラウンドのモデルが信頼性不足で引用拒否の可能性 | 機械検証可能なシグナルが十分、引用確率が高い |

### GEO エコシステムガバナンスの公共利益

すべてのブランドが AI に直接コンテンツを生成させれば、検索エンジンと AI Answer は幻覚コンテンツの闘技場と化します。GEO Rewrite は **「原文事実アンカリング」** の底线を堅持します——原文の事実、データ、観点を変更せず、これらの事実が AI によって発見・引用される構造のみを最適化します。これは本質的に、AI コンテンツ引用エコシステム全体の信頼性を保護することです。

**したがって結論：Rewrite は「事実に基づく信頼性の増幅」、Write は「プロンプトに基づく幻覚生成」です。両者の倫理的比重と責任帰属は根本的に異なります。**

### GEO の終局競争：叙事主権（Narrative Ownership）

GEO の究極の戦場はトラフィック争奪ではなく、**「AI が引用する回答の中で、誰が業界概念を定義するか」** です。

AI がある業界の質問を受けたとき、その定義・フレームワーク・方法論をどこから引用するか——これが AI 時代におけるブランドの認知優先順位を決定します。真の堀はキーワードランキングではなく、AI があなたの業界について語るとき、**まずあなたのブランド定義を連想すること**です。

GEO Rewrite の最終目標は「AI に収録されること」ではなく、**「AI がデフォルトで引用する業界定義ソースになること」** です。

## 使用方法

GEO Rewrite Skill は **自然言語**（初心者向け、直接会話）と **YAML 形式**（精密制御、API・高度なシナリオ向け）の 2 通りの呼び出し方法をサポートします。

### 事前準備（両方式共通）

1. [geo-rewrite-skill/SKILL.md](./geo-rewrite-skill/SKILL.md) を開き、`Ctrl+A` で全選択コピー
2. AI プラットフォームの System Prompt に貼り付け
3. `perspectives/` は不要——v1.2 で視点は全てインライン化

---

### 方式 1：自然言語（初心者推奨）

#### 1. 純粋スコアリング診断 — まずどこが悪いか確認

記事を 6 次元で GEO 品質診断し、スコアレポートのみを出力（リライトなし）：

```
以下のコンテンツの GEO スコアリングを実行してください：
[あなたの記事を貼り付け]
```

**AI の応答例**（完全版には次元スコア、等級、減点理由、原文証拠、最適化提案、推奨視点が含まれます）：

| 次元 | スコア | 等級 | 減点理由 | 原文証拠 |
|------|:--:|------|---------|---------|
| ファクト密度 | 20 | 極弱 | 定量化データなし | 「多くの企業が AI を使い始めている」 |
| 構造明確性 | 25 | 極弱 | 見出しが曖昧 | 「業界動向」 |

総合スコア：19/100（D 級）→ 推奨視点：顧客（データ強化）

> 原文スコア ≥75（A 級）のコンテンツはリライト不要。高スコアコンテンツのリライトをスキップすれば、約 80% のトークンコストを節約可能。

視点を指定してスコアリングの許容度を調整：

```
以下のコンテンツを投資家視点で GEO スコアリングしてください：
[あなたの記事を貼り付け]
```

#### 2. 完全リライト — 診断 + リライト + 比較

```
以下のコンテンツをブランド視点でリライトしてください：
[あなたの記事を貼り付け]
```

**完全リライトの出力内容**：原文 Scorecard → 問題診断 → リライト戦略 → リライト後本文 → FAQ モジュール → Schema 推奨 → リライト後 Scorecard → 比較サマリー → 要補充ソースリスト。

#### 3. 複数視点スタックリライト

```
以下のコンテンツをブランド + 投資家視点でリライトしてください：
[あなたの記事を貼り付け]
```

スタック時、各 DSS+E 次元は全視点中の最高重みを採用。

#### 4. Schema のみ出力

記事の品質が十分で、構造化マークアップだけ追加したい場合：

```
以下のコンテンツの Schema タイプを推奨し、JSON-LD を出力してください：
[あなたの記事を貼り付け]
```

#### 5. AI 引用監査

公開済みコンテンツについて、AI がブランドを正しく理解しているか定期的にチェック：

```
「{あなたのブランド名}」の Oracle Audit プランを生成し、主要 AI プラットフォームでのブランド引用状況をチェックしてください。
```

### 方式 2：YAML 形式（精密制御）

API 呼び出しやバッチ処理など、入力パラメータを精密に制御したい場合に YAML 形式を使用。

#### フィールド説明

| フィールド | 型 | デフォルト | 説明 |
|------|------|--------|------|
| `mode` | rewrite / score_only / schema_only / audit_plan | rewrite | 実行モード |
| `perspective` | 9 種の視点名 | 第三者 | リライト視点 |
| `brand` | テキスト | — | ブランド名 |
| `product` | テキスト | — | 製品名 |
| `audience` | テキスト | — | 対象読者 |
| `source_policy` | use_provided_only / allow_external_sources / mark_missing_sources | mark_missing_sources | ソースポリシー |
| `output_length` | short / standard / long | standard | 出力長さ |
| `need_schema` | true / false | true | Schema 出力の有無 |
| `article` | 複数行テキスト（必須） | — | 処理対象の原文 |

**source_policy 詳細：**

| ポリシー | 意味 | 適用シーン |
|------|------|---------|
| `use_provided_only` | 原文データのみ使用、検索せず、プレースホルダー注釈なし | 原文データが十分、または外部ソース公開不可のコンテンツ |
| `allow_external_sources` | モデルによる検索とソース補充を許可 | モデルが検索機能をサポートする場合 |
| `mark_missing_sources` | 検索しないが、不足箇所に `[要補充ソース：...]` プレースホルダーを付与 | **推奨デフォルト**、安全かつ監査可能 |

#### 例 1：完全リライト

```yaml
mode: rewrite
perspective: 顧客
brand: XX CRM
source_policy: mark_missing_sources
need_schema: true
article: |
  当社の CRM システムは企業の営業効率向上を支援し、多くの顧客から高い評価を得ています。
```

#### 例 2：スコアリングのみ

```yaml
mode: score_only
perspective: 第三者
article: |
  [原文]
```

#### 例 3：Schema のみ

```yaml
mode: schema_only
brand: XX CRM
product: XX CRM 販売管理システム
article: |
  [記事内容]
```

#### 例 4：監査計画

```yaml
mode: audit_plan
brand: XX CRM
article: |
  ブランドの位置づけ、コア製品、主要な方法論などの情報
```

### 診断レポートの読み方

スコアレポートを見るときは「減点理由」列に注目——AI の目から見て、記事のどこが不足しているかを直接教えてくれます：

| 低スコア次元 | 背後にある意味 | AI 引用への影響 |
|---------|-----------|----------------|
| **ファクト密度**低 | 記事に具体的な数字がなく、形容詞に頼っている | AI が検証可能な事実を抽出できず、Embedding 品質が低く、再現率が低い |
| **構造明確性**低 | 結論先行なし、見出しが曖昧、段落が長すぎる | スライス後の各 Chunk の意味が不明瞭、Snippet 抽出失敗 |
| **エンティティ一致性**低 | 同一ブランドに複数の呼称（「XX クラウド」/「XX Cloud」） | AI が安定した Knowledge Graph ノードを構築できず、エンティティリンクが切断 |
| **ソース権威性**低 | 全結論に出典なし、「研究によると」だが具体的ソースなし | AI が低信頼ソースと判断、Reranking で後方に配置 |
| **AI トリガーカバレッジ**低 | 定義文/対比文/因果文/総括文/時効文が全文にない | AI が核心的観点を効率的に抽出できず、Snippet が空または断片化 |
| **Schema 準備度**低 | 純粋な叙述フロー、構造化特徴なし | Schema マークアップがマッチせず、機械解析可能シグナルを喪失 |

---

### 診断結果に基づく視点の選び方

| 最低スコア次元 | 推奨視点 | 理由 |
|-----------|---------|------|
| ファクト密度 + ソース権威性 | **第三者** | データ引用の強化、マルチソースクロス検証 |
| ファクト密度 + 構造明確性 | **顧客** | 定量化成果 + シナリオナラティブ |
| エンティティ一致性 | **ブランド** | ブランド名の統一、エンティティグラフの構築 |
| 全次元低スコア | **投資家** | 全次元最高情報密度要求 |
| ソース権威性 + 構造明確性 | **学術** | 文献引用 + 方法論の透明性 |
| エンティティ一致性 + ソース権威性 | **政府/規制** | 法規引用 + コンプライアンス階層 |

### ソースプレースホルダー使用ガイド

リライト出力の「要補充ソースリスト」には以下のようなマークが出現します：

```
[要補充ソース：業界レポート、推奨ソース：Gartner / IDC / McKinsey]
[要補充データ：2025 年成長率、年範囲：2024-2025]
[要補充事例：顧客業界、使用前後指標、時間周期]
[要補充引用：専門家名、機関、発表日、原文リンク]
```

**これは AI の手抜きではありません——GEO Rewrite の「事実アンカリング」底线が操作レベルで具現化したものです。** AI はここでデータを捏造せず、真実のデータが必要な位置をマークします。ユーザーは公開前に自分の真実のデータでこれらのプレースホルダーを置き換えてください。

## 境界宣言

1. **事実を捏造しない**：既存データ表現のみを強化、データを発明しない。
2. **著者を代替しない**：コンテンツ最適化ツールであり、作成ツールではない。
3. **虚偽の権威シグナルを生成しない**：認証・引用・評価を捏造しない。
4. **法律/コンプライアンス審査を代替しない**：表現構造のみを最適化、法的助言なし。
