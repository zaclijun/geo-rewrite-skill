# GEO Rewrite Skill

[🇨🇳 中文](#) · [🇺🇸 English](./README.en.md) · [🇯🇵 日本語](./README.ja.md)

> 不是替代事实，而是让你已经拥有的真实内容，在 AI 时代被看见。

将传统文章重构为"更容易被生成式 AI 理解、采信、抽取、引用与推荐"的内容。

> 完整 Skill 文件位于 [geo-rewrite-skill/SKILL.md](./geo-rewrite-skill/SKILL.md)（v1.2，25 章节）。

## 核心理念

不是润色文章，而是**让 AI 在生成答案时优先引用你的内容**。

基于 **DSS+E 黄金模型** 驱动改写：

| 维度 | 含义 | 核心要求 |
|------|------|---------|
| **D**epth | 语义深度 | 增强逻辑链条、因果推理、原理机制 |
| **S**upport | 数据支持 | 提升事实密度，优先使用 2025-2026 最新数据 |
| **S**ources | 权威来源 | 多源交叉验证，输出机器可验信号 |
| **E**ntity | 实体权威 | 统一品牌/产品/术语，建立实体图谱 |

## 为什么是 Rewrite 而不是 Write — GEO 生态治理的伦理基础

GEO Rewrite 不是让 AI 替你写，而是让你已经拥有的、真实的内容，在 AI 时代被看见。

### AI 内容生态正在面临"信任危机"

截至 2026 年，互联网上约 27% 的长文内容由 AI 直接生成或辅助生成。当一个生成式 AI 模型以这些 AI 生成的内容作为训练或引用语料时，会产生 **"幻觉递归"（Hallucination Recursion）**——模型引用幻觉内容，产生更多幻觉，导致整个 AI 引用生态的可信度螺旋下降。

**GEO Rewrite 的立场是：不新增事实，只增强事实的可发现性和可引用性。**

### Rewrite vs Write：事实锚定与幻觉生成的本质区别

| | AI Write（直接生成） | GEO Rewrite（改写） |
|---|---|---|
| 事实来源 | 模型训练数据，不可溯源，存在编造风险 | 原文作者的真实信息，可溯源 |
| 数据真实性 | 可能编造具体数字（幻觉） | 仅结构化原文已有数据，不新增任何数值 |
| 品牌实体 | 可能混淆、虚构或张冠李戴 | 原文实体经统一强化，零偏离 |
| 伦理风险 | 幻觉污染引用生态 | 零新增幻觉 |
| AI 引用价值 | 下一轮模型可能因不可信而拒绝引用 | 机器可验证信号充足，采信概率高 |

### GEO 生态治理的公共利益

如果每个品牌都让 AI 直接生成内容，搜索引擎和 AI Answer 将沦为幻觉内容的竞技场。GEO Rewrite 坚持 **"原文事实锚定"** 的底线——不改动原文的事实、数据、观点，只优化这些事实被 AI 发现和引用的结构。这本质是在保护整个 AI 内容引用生态的可信度。

**因此可以得出结论：Rewrite 是"基于事实的可信度放大"，Write 是"基于提示的幻觉生成"。两者的伦理权重和责任归属完全不同。**

### GEO 的终局竞争：叙事主权（Narrative Ownership）

GEO 的终极战场不是流量争夺，而是 **"谁在被 AI 引用的答案中定义行业概念"**。

当 AI 被问到某个行业问题时，它引用的定义、框架、方法论来自谁——这决定了品牌在 AI 时代的认知优先级。真正的护城河不是关键词排名，而是：当 AI 提到你的行业时，**首先联想的是你的品牌定义**。

GEO Rewrite 的最终目标不是"被 AI 收录"，而是 **"成为 AI 默认引用的行业定义源"**。

### 方式一：自然语言（新手推荐）

#### 1. 纯评分诊断 — 先看看文章哪儿不行

对文章进行 6 维度 GEO 质量诊断，只输出评分报告，不改写文章：

```
请对以下内容进行 GEO 评分：
[粘贴你的文章]
```

**AI 返回示例**（完整版包含 6 维度得分、等级、扣分原因、原文证据、优化建议和推荐视角）：

| 维度 | 得分 | 等级 | 扣分原因 | 原文证据 |
|------|:--:|------|---------|---------|
| 事实密度 | 20 | 极弱 | 无量化数据 | "很多企业开始使用 AI" |
| 结构清晰度 | 25 | 极弱 | 标题空泛 | "行业趋势" |

综合评分：19/100（D 级）→ 推荐视角：用户/客户（强化数据）

> 原文评分 ≥75（A 级）的内容无需改写，可直接使用。跳过高分内容的改写可节省约 80% 的 token 成本。

也可以指定视角调整评分的宽容度：

```
请以投资者视角对以下内容进行 GEO 评分：
[粘贴你的文章]
```

#### 2. 完整改写 — 诊断 + 改写 + 对比

```
请以品牌方视角改写以下内容：
[粘贴你的文章]
```

**完整改写会输出**：原文 Scorecard → 问题诊断 → 改写策略 → 改写后正文 → FAQ 模块 → Schema 建议 → 改写后 Scorecard → 对比总结 → 待补充来源清单。

#### 3. 多视角叠加改写

```
请以品牌方 + 投资者视角改写以下内容：
[粘贴你的文章]
```

叠加时各 DSS+E 维度取所有视角中的最高权重。

#### 4. 只出 Schema 建议

当你的文章本身质量不错，只需要添加结构化标记时：

```
请为以下内容推荐 Schema 类型并输出 JSON-LD：
[粘贴你的文章]
```

#### 5. AI 引用审计

已发布内容需要定期检查 AI 是否正确理解了品牌：

```
请为「{你的品牌名称}」生成 Oracle Audit 方案，检查主流 AI 平台对该品牌的引用情况。
```

### 方式二：YAML 格式（精确控制）

当需要精确控制输入参数时（如 API 调用、批量处理），使用 YAML 格式。

#### 字段说明

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `mode` | rewrite / score_only / schema_only / audit_plan | rewrite | 执行模式 |
| `perspective` | 9 种视角名称 | 第三方 | 改写视角 |
| `brand` | 文本 | — | 品牌名称 |
| `product` | 文本 | — | 产品名称 |
| `audience` | 文本 | — | 目标读者 |
| `source_policy` | use_provided_only / allow_external_sources / mark_missing_sources | mark_missing_sources | 来源策略 |
| `output_length` | short / standard / long | standard | 输出长度 |
| `need_schema` | true / false | true | 是否输出 Schema |
| `article` | 多行文本（必填） | — | 待处理的原文 |

**source_policy 来源策略详解：**

| 策略 | 含义 | 适用场景 |
|------|------|---------|
| `use_provided_only` | 仅使用原文中的数据，不搜索，不标注占位符 | 原文数据充足，或内容不允许暴露外部来源 |
| `allow_external_sources` | 允许模型联网搜索补充来源 | 模型支持搜索功能时使用 |
| `mark_missing_sources` | 不搜索，但缺失处标注 `[需补充来源：...]` 占位符 | **推荐默认值**，安全且可审计 |

#### 示例 1：完整改写

```yaml
mode: rewrite
perspective: 用户/客户
brand: XX CRM
source_policy: mark_missing_sources
need_schema: true
article: |
  我们的 CRM 系统帮助企业提高了销售效率，受到很多客户好评。
```

#### 示例 2：仅评分

```yaml
mode: score_only
perspective: 第三方
article: |
  [原文内容]
```

#### 示例 3：只出 Schema

```yaml
mode: schema_only
brand: XX CRM
product: XX CRM 销售管理系统
article: |
  [文章内容]
```

#### 示例 4：审计计划

```yaml
mode: audit_plan
brand: XX CRM
article: |
  你的品牌定位、核心产品、关键方法论等信息
```

### 如何解读诊断报告

查看评分报告时，重点关注「扣分原因」列——它直接告诉你文章在 AI 眼中哪里不够好：

| 低分维度 | 背后的含义 | 对 AI 引用的影响 |
|---------|-----------|----------------|
| **事实密度**低 | 文章缺少具体数字，全文靠形容词支撑 | AI 无法抽取可验证的事实，Embedding 向量质量差，召回率低 |
| **结构清晰度**低 | 无结论先行、标题空泛、段落过长 | AI 切片后每个 Chunk 含义不清，Snippet 抽取失败 |
| **实体一致性**低 | 同一品牌用了多种叫法（"XX云" / "XX Cloud"） | AI 无法建立稳定的 Knowledge Graph 节点，实体链接断裂 |
| **来源权威性**低 | 所有结论都无出处，"研究表明"但无具体来源 | AI 判断为低可信源，Reranking 被后排 |
| **AI 触发覆盖**低 | 全文无定义句/对比句/因果句/总结句/时效句 | AI 无法高效抽取核心观点，Snippet 为空或碎片化 |
| **Schema 就绪度**低 | 纯叙述流，无结构化特征 | Schema 标记无法匹配，丧失机器可解析信号 |

---

### 如何根据诊断结果选择视角

| 最低分维度 | 推荐视角 | 原因 |
|-----------|---------|------|
| 事实密度 + 来源权威 | **第三方** | 强化数据引用，多源交叉验证 |
| 事实密度 + 结构清晰度 | **用户/客户** | 量化结果 + 场景化叙事 |
| 实体一致性 | **品牌方** | 统一品牌名称，建立实体图谱 |
| 全部低分 | **投资者** | 全维度最高信息密度要求 |
| 来源权威 + 结构清晰度 | **学术/科研** | 文献引用 + 方法论透明 |
| 实体一致 + 来源权威 | **政府/监管** | 法规引用 + 合规分级 |

### 来源占位符使用指南

改写输出的「待补充来源清单」中会出现类似标记：

```
[需补充来源：行业报告，建议来源：Gartner / IDC / McKinsey]
[需补充数据：2025 年增长率，年份范围：2024-2025]
[需补充案例：客户行业、使用前后指标、时间周期]
[需补充引用：专家姓名、机构、发布时间、原文链接]
```

**这不是 AI 偷懒——这是 Geo Rewrite 的"事实锚定"底线在操作层面的体现。** AI 不会在这里编造数据，而是标记出需要真实数据的位置。用户应在发布前用自己的真实数据替换这些占位符。

## 边界声明 — 这个 Skill 不做什么

明确边界是对使用者负责，也是对 AI 内容生态负责。

1. **不编造事实**：GEO 评分卡中"事实密度"维度的核心原则是——仅增强原文已有数据的表达能力，不凭空新增原文不存在的数据。凭空增加的增长率、市场份额、ROI 数据被视为"幻觉"，在评分中会被识别并扣分。

2. **不替代作者**：GEO Rewrite 是内容优化工具，不是内容创作工具。原文的事实准确性、观点立场、数据真实性由原文作者负责。工具只负责让这些内容更容易被 AI 理解。

3. **不生成虚假权威信号**：不伪造第三方认证、不虚构行业报告引用、不编造客户评价。所有 Source Authority 维度的评分基于"是否引用了可验证的真实来源"，而非"是否看起来有很多引用"。

4. **不替代法律/合规审核**：在监管合规场景下，本 Skill 仅优化内容表达结构，不提供法律意见。合规声明的准确性需由法务团队审核。

## 扩展性：从单篇 Prompt 到企业级内容管线

GEO Rewrite Skill 的本质是一套结构化 Prompt 规范。任何能够读取文件并调用 AI 模型的工具，都可以将 SKILL.md 作为系统提示词加载，实现规模化改写。

### 本地文件批量改写

| 工具 | 集成方式 |
|------|---------|
| **Claude Code** | 将 SKILL.md 内容写入 `.claude/instructions.md`，通过 `/指令` 或自然语言触发改写 |
| **Trae / Cursor** | 将 SKILL.md 设为项目级 System Prompt（`.trae/rules/` 或 `.cursor/rules/`），在编辑器中选中 `.md` 文件直接改写 |
| **GitHub Copilot Chat** | 通过 `#file:SKILL.md` 引用作为对话上下文，对当前打开的文档执行改写 |
| **Codex CLI** | 通过 `--system-prompt SKILL.md` 参数加载，配合 `--file docs/*.md` 批量处理本地文档 |

### 平台集成批量改写

| 平台 | 集成方式 |
|------|---------|
| **飞书** | 通过飞书自定义机器人 + 应用指令面板加载 SKILL.md，在多维表格或文档中一键触发改写 |
| **Slack** | 通过 Slack Bot 集成，频道内 `@bot geo-rewrite [视角]` 即可改写粘贴的内容 |
| **企业微信** | 通过企微应用 + 知识库接入，将改写能力嵌入企业内部内容审批流 |

### 自动化管线（CI/CD）

```
Git Push → 检测 docs/*.md 变更
        → 调用 AI API + SKILL.md + 指定视角
        → GEO 评分（低于阈值触发改写）
        → PR 提交改写版本
        → 人工审核 → 合并
```

上述所有扩展路径均不需要修改本 Skill 的任何文件——只需将 SKILL.md 作为 System Prompt 注入不同的运行时环境。这就是"规范驱动"的可移植性优势。

- 新视角遵循 DSS+E 权重设计规范，明确四维权重分配
- 提供清晰的适用场景说明和至少一组改写示例
